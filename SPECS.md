# Spécification : VS Code Worktree Handler Plugin

## 1. Vue d'ensemble

Plugin VS Code pour gérer une organisation de repositories multi-worktrees. Permet de basculer entre worktrees tout en maintenant un contexte Git centralisé sur le `.bare/` (bare repository).

**Repo cible** : https://github.com/JoRouquette/worktree-handler

---

## 2. Architecture et structure attendue

### Structure du repository type (exemple KernelAppPlatform)

```
KernelAppPlatform/
├─ .bare/                 # Bare repository (pas de working tree)
├─ main/                  # Worktree (branche main)
├─ release/               # Worktree (branche release)
├─ feature-xyz/           # Worktree (branche feature)
├─ .git                   # Fichier (ref: gitdir: ./.bare)
├─ .gitignore
└─ [autres fichiers racine]
```

### Détection du plugin

- **Activation** : Le plugin se déclenche quand VS Code détecte un `.git` qui pointe vers un `.bare/` interne (`gitdir: ./.bare`).
- **Scope** : Workspace root = dossier parent (KernelAppPlatform/).
- **Bare repo aware** : Le plugin accède toujours à `.bare/` pour les opérations Git de bas niveau (via `git -C .bare/`).

## 3. Fonctionnalités principales

### 3.1 Gestion des worktrees

#### 3.1.1 Détection automatique

- Scanne le répertoire parent au démarrage.
- Liste tous les dossiers au même niveau que `.bare/` qui correspondent à des branches Git existantes.
- Stocke la liste en cache locale avec polling périodique (option : détection au focus de VS Code).

#### 3.1.2 Checkout/Basculement de worktree

- **Commande** : `Worktree: Checkout [branch]`
- **Comportement** :
  1. Met à jour l'explorateur pour afficher uniquement le contenu du worktree actif.
  2. Marque visuellement le worktree courant dans l'UI du plugin.
  3. Génère/met à jour le script sourçable pour le terminal (`.worktree-setup.sh` / `.worktree-setup.ps1`).
  4. Sauvegarde l'état du worktree courant dans `.vscode/.worktree-state`.
- **Read-only pour autres worktrees** (en mode grisé) : Les fichiers des worktrees non-actifs sont marqués en lecture seule dans VS Code.
  - Implémentation : Workspace settings ou trust model.
  - Objectif : Éviter les modifications accidentelles dans le mauvais contexte.
- **Feedback utilisateur** : Toast de confirmation, statusbar mise à jour.

#### 3.1.3 Visibilité des autres worktrees (configurable)

- **Mode 1 (Masquage complet)** : Seul le worktree actif apparaît dans l'explorateur.
- **Mode 2 (Grisé + Read-only)** : Les autres worktrees apparaissent grisés/désactivés et en lecture seule.
- **Stockage** : Setting VS Code `worktreeHandler.otherWorktreesVisibility` (`hidden` | `dimmed`).

---

### 3.2 Explorateur et contexte Git

#### 3.2.1 Vue Explorateur

- **Racine logique** : Le dossier du worktree courant (ex: `main/`).
- **Fichiers cachés** : `.bare/` est toujours caché (jamais visible même avec "Show Hidden Files").
- **Filtrage** : Les autres worktrees peuvent être masqués ou grisés selon la config.

#### 3.2.2 Git et Source Control

- **Contexte Git persistant** : Le contexte Git reste pointé sur `.bare/` (repo logique).
- **Détection de branche** : Utilise le **nom du dossier du worktree courant** comme branche affichée dans la statusbar (ex: folder `main/` → branche `main`).
  - Rationale : Le `.bare/HEAD` peut pointer vers une branche différente (ex: `develop`) et n'est pas représentatif du contexte de travail réel.
  - Simple et fiable : Le nom du worktree = la branche checked out dans ce contexte.
- **Historique et diff** : Fonctionnent nativement contre `.bare/` sans modification du plugin.
- **Limitation acceptée** : Les opérations "Sync" ou "Publish" de VS Code Git ne seront pas customisées.

---

### 3.3 Terminal intégré

#### 3.3.1 Configuration minimale

Le plugin ne modifie **aucun setting VS Code** pour le terminal. À la place :

1. Génère un script sourçable (`.worktree-setup.sh` ou `.worktree-setup.ps1`) dans le repo root.
2. L'utilisateur source ce script manuellement ou l'ajoute à son `.bashrc`/`.zshrc`/`.profile`.
3. Le script lit l'état du plugin (worktree courant) et se configure en conséquence.

#### 3.3.2 Script sourçable

**Exemple `.worktree-setup.sh`** :

```bash
#!/bin/bash
# Auto-generated by worktree-handler plugin
# Source this in your shell: source .worktree-setup.sh

WORKTREE_ROOT=$(pwd)
CURRENT_WORKTREE=$(cat "$WORKTREE_ROOT/.vscode/.worktree-state" 2>/dev/null || echo "main")

if [ -d "$WORKTREE_ROOT/$CURRENT_WORKTREE" ]; then
  cd "$WORKTREE_ROOT/$CURRENT_WORKTREE"
  export GIT_DIR="$WORKTREE_ROOT/.bare"
  export GIT_WORK_TREE="$(pwd)"
else
  echo "[worktree-handler] Warning: worktree '$CURRENT_WORKTREE' not found"
fi
```

#### 3.3.3 Mise à jour du script

- Le plugin met à jour ce script chaque fois qu'un worktree est basculé.
- L'utilisateur re-source le script ou ouvre un nouveau terminal.
- Optionnel : le plugin affiche un toast `Worktree setup script updated, run: source .worktree-setup.sh`.

#### 3.3.4 Stockage d'état

- Chemin : `.vscode/.worktree-state` (fichier texte simple, contient le nom du worktree courant).
- Lisible par le script et par le plugin.
- Ajouté à `.gitignore` du `.bare/` (repo level).

#### 3.3.5 Support multi-shell

- `.worktree-setup.sh` pour Bash, Zsh.
- `.worktree-setup.ps1` pour PowerShell (si détecté).
- Le plugin génère les deux versions automatiquement.

## 4. Interface utilisateur

### 4.1 Palette de commandes

- `Worktree: Checkout [branch]` – Liste déroulante des branches/worktrees détectés.
- `Worktree: Refresh List` – Force la détection des worktrees.
- `Worktree: Show Configuration` – Affiche les settings actuels.

### 4.2 Statusbar

- **Élément** : Badge/text affichant le worktree courant (ex: `$(git-branch) main`).
- **Click** : Ouvre la palette "Worktree: Checkout".
- **Mise à jour** : À chaque basculement, au focus de VS Code, ou polling périodique.

### 4.3 Arborescence dédiée (Optional)

- **Panneau latéral** : Liste les worktrees avec icône (branche active = highlight).
- **Click** : Bascule vers ce worktree.
- **Refresh** : Icône pour rescanner les branches.

## 5. Configuration et persistance

### 5.1 Settings (settings.json)

```json
{
  "worktreeHandler.enable": true,
  "worktreeHandler.otherWorktreesVisibility": "hidden",
  "worktreeHandler.autoDetectWorktrees": true,
  "worktreeHandler.pollInterval": 5000,
  "worktreeHandler.rememberLastWorktree": true,
  "worktreeHandler.generateTerminalScripts": true
}
```

### 5.2 État local (Global storage)

- Dernier worktree ouvert.
- Cache des worktrees détectés + timestamps.

### 5.3 Workspace storage (.vscode/settings.json)

- Overrides des settings au niveau du workspace.
- `.vscode/.worktree-state` : État du worktree courant (lu par scripts).

## 6. Gestion des cas limites

### 6.1 Pas de `.bare/` détecté

- Plugin reste inactif.
- Pas d'erreur, comportement standard VS Code.

### 6.2 `.git` ne pointe pas vers `.bare/`

- Plugin ignore le workspace.

### 6.3 Dossier vide ou inexistant

- Affiche un message d'avertissement et désactive les commandes.

### 6.4 Worktree supprimé manuellement

- Plugin rescanne automatiquement.
- Si le worktree courant est supprimé → bascule vers un worktree valide ou affiche une alerte.

### 6.5 Édition de fichiers dans plusieurs worktrees

- VS Code gère nativement ; les fichiers ouverts restent dans leur worktree d'origine.
- Pas de "sync" automatique entre worktrees.

## 7. Architecture technique (TODO pour l'implémentation)

### 7.1 Stack

- **TypeScript** + VS Code Extension API.
- **Dependencies** : `simple-git` ou `@vscode/git-api` pour git ops.
- **Bundling** : esbuild ou webpack.

### 7.2 Modules principaux

1. **WorktreeManager** – Détection, scan, basculement.
2. **GitContext** – Interaction avec `.bare/` et métadonnées.
3. **UI Controller** – Statusbar, commandes, palette.
4. **TerminalScriptHandler** – Génération des scripts sourçables.
5. **StateManager** – Persistence (global + workspace storage).

### 7.3 Workflow d'activation

1. Extension activée sur `onView:worktreeHandler` ou au `onStartupFinished`.
2. Détecte `.git` → `gitdir: ./.bare` au workspace root.
3. Initialise les modules, restaure le dernier worktree.
4. Écoute les événements : focus, changedConfiguration, fileSystemWatcher (`.bare/`).
